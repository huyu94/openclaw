OpenClaw 与 Pi：关于编程智能体的一些思考


如果你最近关注科技圈，一定注意到了我朋友 Peter 的项目本周在网上爆火。它有很多名字，最新的叫 OpenClaw，但根据你看到报道的时间，你可能在新闻里见过它叫 ClawdBot 或 MoltBot。它是一个连接到你常用通讯渠道的智能体（Agent），核心功能只有一个：运行代码。



但你可能不知道的是，OpenClaw 的底层是一个叫 Pi 的小型编程智能体。Pi 目前几乎是我唯一在用的编程智能体，过去几周，我彻底成了这个小工具的“专家”。在最近做完一场相关演讲后，我意识到还没在博客上写过它，所以有必要聊聊为什么我对它如此痴迷，以及它与 OpenClaw 的渊源。



Pi 由 Mario Zechner 开发。与 Peter 追求的“带点疯狂的科幻感”不同，Mario 极其务实。尽管风格迥异，OpenClaw 和 Pi 都遵循同一个理念：大语言模型（LLM）极其擅长编写和运行代码，所以要彻底拥抱这一能力。 某种程度上这并非偶然——因为是 Peter 在去年带我和 Mario 掉进了这个“智能体”的坑。



Pi 是什么？


Pi 是一个编程智能体。老实说，现在的编程智能体泛滥成灾，随便挑一个都能体验“智能体化编程”。我之前在博客里高度评价过 AMP，原因之一是它给我的感觉是：开发者自己就在沉迷代理编程，亲手验证过什么方案才真正好用，而不是只做了个花哨的界面。



Pi 吸引我有两个核心原因：



极简核心： 它的系统提示词是我见过最短的，只提供四个核心工具：Read（读）、Write（写）、Edit（改）、Bash（执行）。
扩展系统： 它通过扩展系统弥补了核心的精简，而且扩展能将状态持久化到会话中，这一功能极其强大。
额外加分项： Pi 本身写得非常有质感。界面不闪烁，内存占用低，运行极其稳定——这显然出自一个对软件品质有近乎偏执追求的人之手。



Pi 也是一组组件的集合，你可以在此基础上构建自己的智能体。OpenClaw 就是这么做的，我自己的 Telegram 机器人也是，Mario 给他妈妈做的那个也是。如果你想做一个连接到特定平台的智能体，只要把 Pi 指向它自己，它就能帮你搭出一个原型。



Pi 舍弃了什么？


要理解 Pi，更重要的是理解它故意不做什么。



最明显的例子：它不支持 MCP（模型上下文协议）。虽然你可以写扩展来支持它，但 OpenClaw 的做法更有意思——它利用 mcporter 将 MCP 调用转换为 CLI 接口或 TypeScript 绑定，然后让智能体按需使用。

这不是因为懒，而是 Pi 的核心哲学：如果智能体不会做某件事，你不应该去下载别人的扩展或技能，而是让智能体自己写代码来扩展自己。 Pi 崇尚的是“代码生成代码、代码运行代码”的闭环。

当然，你也可以下载别人的扩展。但 Pi 更鼓励你这样做：把既有扩展丢给它，然后说：“照着这个写，但改成我想要的样子。”



为“智能体造智能体”而设计


观察 Pi（以及 OpenClaw）的运作方式，你会发现它像黏土一样具有极强的可塑性。这对底层架构提出了特殊要求：

跨模型兼容： Pi 的 AI SDK 支持在一个会话中包含来自不同供应商的消息。它深知会话在不同厂商间的迁移成本，因此不会过度依赖某个厂商的特定功能。
持久化状态： 除了模型消息，它还在会话文件中维护自定义元数据。扩展可以用它存状态，系统也可以存那些不发给 AI（或只发一部分）的信息。
热重载与闭环： 因为状态能存盘，Pi 支持热重载。智能体可以编写代码、重载系统、测试、迭代，直到扩展可用。它自带文档和示例，方便智能体“自学”如何扩展自己。
树状会话结构： 这一点至关重要。你可以开启分支，在会话中自由导航。比如你可以开个“支线”去修一个坏掉的工具，修复后再回退到主线，Pi 会自动总结支线发生了什么。这完美解决了 MCP 的痛点——在大多数平台，你很难在不重置上下文的情况下动态改变工具能力。


上下文之外的工具


Pi 的扩展确实可以向 LLM 注册工具。虽然我通常更偏爱 CLI，但在某些场景下，工具化更合适。例如，我让智能体为我写了一个本地任务追踪器，我给它提供了一个待办清单工具，这是目前我唯一加载到上下文中的额外工具。

但我大部分时间都在用扩展来提升“幸福感”。Pi 的扩展可以直接在终端渲染 TUI（文本用户界面）组件：进度条、交互式文件选择器、数据表、预览面板等。Mario 甚至证明了可以在里面运行 Doom——虽然没啥用，但这意味着你可以轻松搭出精美的调试仪表盘。



以下是我常用的一些扩展，核心理念依然是：让智能体根据你的习惯去改造它们。

/answer： 我不喜欢预设的“计划模式”，我喜欢和智能体有来有回。当智能体提出一堆问题时，/answer 会提取最后一条回复中的问题，并生成整洁的输入框供我填写。
/todos： 以 Markdown 形式存储在 .pi/todos。我和智能体都能操作，它能自动领任务并标记进度。
/review： 在代码交付给我之前，智能体会先在分支会话中进行自审。界面参考了 Codex，可以方便地查看 diff 或远程 PR，并根据我的偏好（如关注新依赖）给出提醒。
/files： 管理会话中引用的文件。支持在 VS Code 中比对或快速预览，生成 PDF 时尤其好用。


软件构建软件


这些扩展的重点在于：它们不是我写的，而是智能体根据我的需求自己写的。



我告诉 Pi“写个扩展”，它就搞定了。不需要满世界找 MCP 插件或社区技能，即便我用了一些现成的，也是经过智能体亲手改造的。例如，我把所有复杂的浏览器自动化工具都换成了智能体自己用 CDP（Chrome DevTools 协议）写的简单技能。



与 Pi 这种极简智能体协作的迷人之处在于：它让你真正实践了“用软件构建更多软件”。将其推向极致，就是隐去所有 UI，直接连接到你的聊天工具——这便是 OpenClaw。看着它的爆发式增长，我深感：无论形式如何，这就是开发的未来。
